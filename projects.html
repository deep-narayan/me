Overview

The Mahakumbh ATM project was one of the most technically ambitious initiatives I've led — an end-to-end system connecting IoT hardware, cloud software, and a government analytics SaaS, built for one of the world's largest human gatherings: Mahakumbh 2025 in Prayagraj, Uttar Pradesh.

The system managed 150 cloud-connected water dispensers, enabling pilgrims to access clean water through QR-based payment or free access (controlled remotely). Daily water consumption reports were generated and submitted directly to the Prime Minister's Office (PMO).

Architecture

System Architecture
Dispenser Hardware (C++) → MicroPython Controller → AWS EC2 (Windows VM)
Django Backend API → PostgreSQL DB + Razorpay Payment API
PySpark Analytics → SaaS Dashboard (UPJN) → PMO Reports

Hardware Layer

The physical dispensers were upgraded to include a user-friendly display panel capable of showing QR codes for payments. I led the hardware integration, programming dispenser firmware in C++ and MicroPython to communicate over Wi-Fi with the cloud backend.

Software & Backend

  • Built a Django REST API as the central command layer for all 150 dispensers — enabling real-time remote configuration, status monitoring, and dynamic control.
  • Integrated Razorpay API to generate unique QR codes per dispenser, enabling seamless cashless transactions for water access.
  • Implemented cloud-based commands allowing the UPJN team to switch dispensers to "free mode" during specific periods — crucial for large-scale crowd management.
  • Deployed the entire backend stack on AWS EC2 (Windows VM) for reliability and scalability during peak pilgrimage periods.

SaaS & Analytics

  • Developed a PySpark-powered analytics SaaS for the UPJN team — providing tap-level, machine-level, and zone-level water consumption data.
  • Automated generation of daily water consumption reports submitted to the PMO, ensuring full accountability and government oversight.
  • The system consistently delivered 1.5+ lakh litres of water per day throughout the event.

Key Challenges

  • Ensuring 99%+ uptime for 150 devices operating in challenging outdoor conditions at a large public event.
  • Real-time remote configuration of devices — enabling free water periods without physical access to hardware.
  • Scaling PySpark jobs to process high-frequency sensor data in near real-time for government reporting.

Overview

A full-featured Customer Relationship Management platform built from scratch for Aster Technologies. The CRM handles the complete customer lifecycle — lead capture, pipeline management, communication tracking, task assignment, and reporting.

Key Features & Technical Decisions

  • Django + PostgreSQL Backend: Chose Django for its ORM power and rapid development capabilities. Implemented optimised ORM queries with select_related/prefetch_related, eliminating N+1 issues and reducing query count by 60%.
  • React.js Frontend: A component-driven React dashboard with hooks-based state management. Real-time activity feed powered by WebSocket (Django Channels).
  • Redis Caching: Implemented Redis for session management and frequently accessed data — reducing average page load times by 20%.
  • Role-Based Access Control: Custom RBAC system with granular permissions per user type (Admin, Manager, Agent).
  • Celery + Redis: Async task queue for email notifications, report generation, and scheduled follow-up reminders.
  • Docker + AWS EC2: Containerised deployment with Docker Compose, hosted on AWS EC2 with NGINX reverse proxy and SSL.

Business Impact

The CRM directly contributed to a 15% increase in sales revenue by centralising customer data, automating follow-up workflows, and giving the sales team a unified view of their pipeline. Page load optimisations improved user experience significantly.

Overview

A comprehensive HR automation platform that replaced all manual HR processes at Aster Technologies. The system handles the complete employee lifecycle from daily attendance to monthly payroll, all automated and auditable.

Modules

  • Attendance Management: Automated real-time attendance tracking, eliminating manual punch-in records. Integrated with employee devices for seamless logging.
  • Leave Management + Slack Bot: Built a custom Python Slack bot that handles leave requests end-to-end — employees request leave via Slack, managers approve/reject via bot buttons, and the system auto-updates attendance records.
  • Payroll Automation: Complex payroll engine built in Python handling salary computation, deductions, PF/ESI calculations, and tax logic. Fully automated monthly payroll run via Celery scheduled tasks.
  • Salary Slip Email Automation: Auto-generates professional PDF salary slips and emails them to each employee every month — zero manual intervention required.

Why Waterfall?

Unlike the Agile-managed CRM, this project used the Waterfall methodology due to the sequential dependency of HR modules — payroll depends on attendance, which depends on leave data. Clear upfront requirements made Waterfall the right choice here.

04 JAVA · SPRING BOOT

Inventory Microservice

Backend Developer · Java / Spring Boot
Architecture
Client / API Consumer → Spring Boot REST API → MySQL (Hibernate ORM)
Spring Security + JWT + Maven Build + Swagger / OpenAPI Docs

Overview

A production-ready RESTful microservice for inventory management, built with Spring Boot — my entry into Java-based enterprise backend development. This project demonstrates service-oriented architecture patterns used in large-scale financial and enterprise systems.

Technical Implementation

  • Spring Boot 3.x: Built with auto-configured Spring MVC for REST endpoints — clean, annotation-driven controllers with proper separation of concerns (Controller → Service → Repository layers).
  • Hibernate + JPA: ORM-based data access with lazy loading, transaction management, and custom JPQL queries for complex inventory aggregations.
  • Spring Security + JWT: Stateless JWT authentication with role-based access — Admin can perform full CRUD, Viewer has read-only access.
  • Pagination & Filtering: Spring Data Pageable implementation for efficient large dataset handling.
  • OpenAPI / Swagger: Auto-generated, interactive API documentation with Springdoc.
  • Maven: Dependency management and build lifecycle — structured multi-module project layout.

Learning & Comparison with Python

Building this in Java gave me a deep appreciation for Spring Boot's dependency injection and its strong typing system. Compared to Django, Spring Boot requires more boilerplate but offers greater compile-time safety and is a standard in enterprise environments like banking and fintech — which is a key reason I'm building this skill.

05 REACT

Analytics Dashboard — React.js

Frontend Developer · React.js + REST APIs
Component Architecture
App.jsx (Root) → Dashboard Layout → Chart / Table Components

Overview

An interactive data analytics dashboard built with React.js, designed to consume REST APIs and display dynamic data visualisations with real-time polling. This project was built to extend my frontend capabilities beyond HTML/CSS templates and into modern component-driven architecture.

React Concepts Applied

  • Functional Components + Hooks: All components use functional style. useState for local state, useEffect for API side-effects, and custom useFetch hook for reusable data fetching logic.
  • Component Composition: Decomposed the dashboard into KPICard, LineChart, DataTable, FilterBar, and DateRangePicker components — each independently testable and reusable.
  • Real-Time Data Polling: useEffect-based polling mechanism with configurable intervals — updates charts without page reload.
  • Chart.js Integration: Responsive line, bar, and donut charts with animated transitions on data update.
  • CSS Modules: Scoped styling to prevent class collisions across components.
  • Error & Loading States: Graceful handling of API failures with skeleton loaders and error boundary components.

Connection to Backend Work

Having built Django REST APIs on the backend, building this React frontend gave me end-to-end understanding of the full request lifecycle — from database query optimisation to JSON serialisation to React state rendering. This full-stack perspective makes me a more effective developer in both directions.

06 ML · NLP

Sentiment Analysis & NLP Pipeline

ML Engineer · Python · NLTK · spaCy · Scikit-learn
ML Pipeline
Raw Text Input → Preprocessing (NLTK) → TF-IDF Vectorisation
Scikit-learn Classifier → spaCy NER → Flask API (Predictions)

Overview

A text classification and sentiment analysis pipeline built to explore the fundamentals of Natural Language Processing and Machine Learning with Python. The pipeline processes raw text, extracts features, trains classifiers, and serves predictions via a Flask API.

NLP Pipeline Stages

  • Text Preprocessing (NLTK): Tokenisation, stop-word removal, stemming/lemmatisation, and punctuation cleaning using NLTK pipelines.
  • Feature Engineering — TF-IDF: Term Frequency-Inverse Document Frequency vectorisation to convert text into numerical feature matrices. Experimented with n-gram ranges (unigrams + bigrams).
  • Classification (Scikit-learn): Trained and compared multiple classifiers — Logistic Regression, Naive Bayes, and SVM. Evaluated using precision, recall, F1-score, and confusion matrix.
  • Named Entity Recognition (spaCy): Extracted people, organisations, locations, and dates from unstructured text using spaCy's pre-trained en_core_web_sm model.
  • Sentiment Analysis: Polarity classification (positive/negative/neutral) using a combination of VADER (rule-based) and ML-based approaches — compared performance of both.
  • Flask API: Wrapped the trained pipeline in a REST API accepting POST requests with raw text and returning predictions with confidence scores.

Key Learnings

  • Hands-on understanding of the difference between rule-based NLP (VADER, regex) and statistical ML approaches.
  • Practical experience with the bias-variance tradeoff through hyperparameter tuning.
  • Understanding of vectorisation strategies — bag-of-words vs TF-IDF vs word embeddings (explored Word2Vec basics).
  • How to productionise ML models — serialising with joblib, versioning models, and serving via REST API.

Next Steps

Planning to extend this into transformer-based approaches using HuggingFace Transformers (BERT fine-tuning) for significantly improved accuracy on domain-specific text classification tasks.
